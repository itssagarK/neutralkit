How Neutralinojs Works Internally

Every API in this proposal follows the same pattern: a thin JavaScript wrapper sends a JSON call over WebSocket IPC to a C++ handler, which delegates directly to the OS. No intermediate libraries, no runtime overhead, ~1MB binary.

Design Principles · All 6 Proposed APIs
📦
Zero New Dependencies
Every proposed API uses only platform stdlib or Win32/POSIX headers already present in the build. No CMakeLists.txt changes. No third-party code. Bundle stays at ~1MB.
stdlib · Win32 · POSIX — already in build
🔧
Consistent Code Pattern
Every API follows the same 4-step pattern proven by PR #1632. One router entry, one .h declaration, one .cpp implementation with platform guards, one JS export.
router → os.h → os.cpp → neutralino.js
🖥
Cross-Platform Guards
All platform code uses #ifdef _WIN32 / #elif __APPLE__ / #else guards — the exact same pattern used in 12+ places in os.cpp already. No new patterns introduced.
#ifdef _WIN32 · #elif __APPLE__ · #else
🔑
OS Delegation Principle
Neutralinojs stays lightweight by delegating to the OS instead of bundling implementations. gethostname() is one OS call — not a 50KB library. This is the framework's core philosophy.
gethostname() → 1 syscall · 0KB added
🧪
Test Coverage First
Every new API gets spec tests in spec/os.spec.js or spec/computer.spec.js covering type checking, valid range validation, and edge cases. PR #1608 proved this works.
spec/os.spec.js · type + range + edge cases
⚡
Novel: Zero-Poll Clipboard Events
os.watchClipboard() uses native OS event push mechanisms — no polling timer. First desktop framework to do this. Electron and Tauri both still require setInterval() workarounds.
AddClipboardFormatListener() · WM_CLIPBOARDUPDATE
